# TMGG Codebase Audit — 2026-02-18

This audit evaluates the TMGG codebase across five dimensions: mathematical correctness, code architecture, researcher extensibility, documentation quality, and generative experiment observability. The codebase supports graph denoising experiments and DiGress-style discrete diffusion generative modeling; the audit focuses on whether the implementation is correct, maintainable, and extensible for researchers wanting to add architectures, datasets, and run experiments locally or on Modal.

Findings are grouped by severity within each section. No changes were made — this document is diagnostic only.

---

## 1. Mathematical Correctness and Naming

### Critical

**1.1 KL prior direction is reversed in the VLB**
`src/tmgg/experiments/discrete_diffusion_generative/lightning_module.py:583-596`

`_kl_prior` calls `F.kl_div(input=probX.log(), target=limit_dist_X)`, which computes KL(limit_dist || q(z_T|x_0)) — the reverse of the standard VLB term KL(q(z_T|x_0) || p(z_T)). A comment says "we keep this to match the original DiGress implementation," but the original DiGress code computes the correct direction using `(q * (q.log() - p.log())).sum(-1)` directly. For well-tuned schedules the value is near zero, so training impact is negligible, but logged `val/kl_prior` values are technically wrong. Fix: swap the arguments to `F.kl_div`.

**1.2 `E[..., 0]` used as adjacency matrix (should be `E[..., 1]`)**
`src/tmgg/models/digress/transformer_model.py:420-422, 732-733, 740`

When `use_gnn_projections` is active, the code falls back to `A = E[..., 0]`. In the categorical DiGress representation, class 0 is "no edge" and class 1 is "edge." `E[..., 0]` is the probability of *no* edge — the logical complement of an adjacency matrix. Every GNN graph-convolution layer in the discrete diffusion path receives an inverted adjacency, computing message-passing on non-edges. The same issue affects spectral eigenvector extraction at line 740.

**1.3 `p_in`/`p_out` config keys silently ignored — SBM probabilities not configurable**
`src/tmgg/exp_configs/base_config_discrete_diffusion_generative.yaml:56-57` vs `src/tmgg/experiment_utils/data/multigraph_data_module.py:134-135`

The discrete diffusion config uses `p_in: 0.7` / `p_out: 0.1`, but the code reads `p_intra` / `p_inter` with `.get(..., default)`. The config values are silently ignored; the code always uses the hardcoded defaults (which happen to be 0.7/0.1). You cannot actually change SBM edge probabilities in the discrete diffusion experiment via config. The Gaussian diffusion config correctly uses `p_intra`/`p_inter`.

**1.4 `s_int = -1` at `t=0` wraps around `alphas_bar` via negative indexing**
`src/tmgg/experiments/discrete_diffusion_generative/lightning_module.py:287-293`

During training with `lowest_t=0`, `t_int` can be 0, producing `s_int = -1` and `s_float = -1/T`. Then `alpha_s_bar = self.noise_schedule.get_alpha_bar(t_normalized=-1/T)` indexes `alphas_bar[-1]` — the fully noisy value rather than 1.0 (clean). This bad value is stored but only consumed during validation (where `lowest_t=1`), so it is not a live training bug, but it is a latent correctness hazard.

### Naming and Terminology

**1.5 Four names for two SBM parameters**: `p_in`/`p_out` (discrete config), `p_intra`/`p_inter` (Gaussian config, data modules), `p`/`q` (internal `generate_sbm_adjacency`). Standardize on `p_intra`/`p_inter`.

**1.6 `PlaceHolder` is the primary data container, not a placeholder.** Used throughout the diffusion pipeline for batched graph data, transition matrices, posterior distributions, and model outputs. Name suggests a temporary stub. Should be `GraphFeatureBatch` or similar. (`diffusion_utils.py:10-45`)

**1.7 `Xtoy`/`Etoy` read as "toy" rather than "X-to-y."** Should be `NodeToGlobal`/`EdgeToGlobal`. (`layers.py:7-44`)

**1.8 `loss_term_0` / `_reconstruction_loss` is actually a log-probability** (positive contribution that reduces NLL), not a loss. The method name and variable name contradict the semantics. (`lightning_module.py:450-462`)

**1.9 `"spectre"` metric key vs `spectral_mmd` logged key.** Config enables the metric with `"spectre"` but it logs as `spectral_mmd`. The original "SPECTRE" is a dataset/model name (Martinkus et al. 2022), repurposed here as a metric name for Laplacian eigenvalue MMD. (`sampling_evaluator.py:81`, `mmd_metrics.py:44`)

**1.10 `epoch_NLL` uses uppercase `NLL`** while `kl_prior`, `kl_diffusion`, `reconstruction_logp` are lowercase. Inconsistent W&B key casing. (`lightning_module.py:484`)

**1.11 `Qsb` notation unexplained.** `sb` means "s-bar" (Q_{s|0}, the cumulative transition), but reads as "Q from s to b." (`diffusion_utils.py:376-405`)

**1.12 `DigressNoiseGenerator.eps` convention inverted.** `eps=0` means clean, `eps=1` means fully noisy — opposite of other `NoiseGenerator` subclasses where `eps` increases noise monotonically. (`noise_generators.py:69-89`)

**1.13 `_val_nll`/`_val_reconstruction` accumulators reused for test phase.** The `_val_*` prefix misleads readers encountering these in `test_step`. (`lightning_module.py:153-157, 416-427`)

**1.14 Dead Gaussian diffusion utilities mixed with active code.** `gaussian_KL`, `SNR`, `sigma`, `alpha`, etc. in `diffusion_utils.py:172-262` are only used by the stubbed `ContinuousDiffusionLightningModule`. Their presence alongside active discrete diffusion code creates confusion.

## 2. Code Architecture and Design

### Architectural Issues

**2.1 Two parallel Lightning module hierarchies with no shared base.**
`DenoisingLightningModule` (`base_lightningmodule.py:37`) is the base for all denoising experiments. `DiscreteDiffusionLightningModule` (`discrete_diffusion_generative/lightning_module.py:48`) inherits directly from `pl.LightningModule` — entirely separate. No shared abstract base for "models that do graph generation." A researcher adding a new generative architecture must copy ~900 lines of VLB math.

**2.2 `_make_model` template method is a design trap.**
`DenoisingLightningModule.__init__` calls `self._make_model()` during `super().__init__()`, before the subclass `__init__` body runs. Two incompatible workarounds exist across subclasses: some pass model parameters through `**kwargs` to super, others call `save_hyperparameters()` before `super().__init__()`. Inconsistency means new subclasses will copy the wrong pattern ~50% of the time. (`base_lightningmodule.py:236-258`)

**2.3 `configure_optimizers` duplicated identically** between `DenoisingLightningModule` and `DiscreteDiffusionLightningModule`, because the generative module cannot inherit from the denoising base. (`base_lightningmodule.py:156-169`, `discrete_diffusion_generative/lightning_module.py:163-176`)

**2.4 MMD epoch-end logic duplicated in 2-3 places.** `on_validation_epoch_end` / `on_test_epoch_end` in both `DigressDenoisingLightningModule` and `DiscreteDiffusionLightningModule` contain near-identical accumulate-sample-evaluate-log patterns. (`digress_denoising/lightning_module.py:116-181`, `discrete_diffusion_generative/lightning_module.py:389-460`)

**2.5 Model registration does not cover generative architectures.** `MODEL_REGISTRY` and `create_model()` handle all denoising models cleanly. Generative architectures (`DiscreteGraphTransformer`) are instantiated via Hydra's `_target_` mechanism — a different pattern entirely. (`factory.py` vs `discrete_default.yaml:18`)

### Redundancy

**2.6 `masked_softmax` defined twice** — identical logic in `models/digress/layers.py:47` and `models/layers/masked_softmax.py:6`.

**2.7 `GNN` and `GNNSymmetric` share 90% of their bodies.** The only difference is symmetric vs asymmetric output projections. Additionally, `GNN.embeddings()` duplicates the first two-thirds of `GNN.forward()`. (`gnn.py`, `gnn_sym.py`)

**2.8 Train/val/test split logic implemented three times** — `multigraph_data_module.py:183-194`, `synthetic_graphs.py:612-621`, `pyg_datasets.py:196-203`. All could use a single utility.

**2.9 Six near-identical runner files.** Each `experiments/*/runner.py` is ~33 lines: a `@hydra.main` decorator with a different `config_name` and a call to `run_experiment(cfg)`. Could be a single parameterized runner.

### Separation of Concerns

**2.10 `run_experiment.py` inspects model type with `hasattr`** to decide whether to call `final_eval`. This encodes model-type knowledge in the orchestration layer. (`run_experiment.py:127-148`)

**2.11 `logging.py:create_loggers` is 190 lines** mixing credential checking, S3 env config, automatic logger creation, and `setattr`-based metadata attachment on third-party objects (`_s3_sync_path` on `TensorBoardLogger`). The contract between `create_loggers` and `sync_tensorboard_to_s3` is invisible to the type system.

**2.12 `base_lightningmodule.py` `_log_visualizations` is 115 lines** of matplotlib/logging code inside the training module, directly accessing dataloaders and running model inference.

### Brittle Patterns

**2.13 Inconsistent default split ratios.** `MultiGraphDataModule`: 0.8/0.1; `GraphDataModule`: 0.6/0.2; `SyntheticGraphDataset`: 0.7/0.1; `PyGDatasetWrapper`: 0.7/0.1. Four different defaults for the same conceptual operation.

**2.14 `SyntheticGraphDataset.__init__` is a 60-line if-elif dispatch chain.** Same problem the `MODEL_REGISTRY` pattern was designed to solve. (`synthetic_graphs.py:527-557`)

**2.15 Magic numbers.** `seed + 1000` for test seed derivation (`single_graph_data_module.py:308`); `min(5, ...)` for max validation partitions (`multigraph_data_module.py:294`); factor `4` in `Xtoy` linear layer (`layers.py:13`); `max_nodes=200` hardcoded default in several models (`factory.py:103,297`).

## 3. Researcher Extensibility

### High Friction

**3.1 Adding a new denoising architecture requires 5-6 touch points:** (1) implement `DenoisingModel` subclass, (2) register in `factory.py`, (3) create model YAML config, (4) decide which `LightningModule` to use or create new one, (5) add `pyproject.toml` entry point and base config if new module, (6) optionally add Modal stage definition. Steps 1-2 are clean; 3-6 involve boilerplate with no scaffolding or docs.

**3.2 Adding a new DiGress generative backbone** follows a completely different pattern (Hydra `_target_` instantiation, model passed as constructor argument) than denoising (factory pattern, `_make_model` template method). A researcher reading only the factory pattern would be lost.

**3.3 Two incompatible data pipelines.** Denoising uses adjacency matrices via `GraphDataModule`; discrete diffusion uses categorical one-hot features via `SyntheticCategoricalDataModule`. A researcher cannot test the same graph distribution with both experiment types without implementing the data pipeline twice.

**3.4 Hydra config inheritance is 5 levels deep** for a typical denoising run. When a value doesn't override as expected, the error is an OmegaConf `InterpolationResolutionError` pointing to the interpolation site, not the missing key. The `@package _global_` directive in `base/callbacks/default.yaml` puts callbacks at the root namespace — non-obvious and conflicting with the `@package data` convention.

**3.5 `noise_levels` propagated three different ways.** Global `noise_levels` → `data.noise_levels` (via interpolation) → `model.noise_levels` (re-interpolated). Spectral model configs read `${noise_levels}` directly; GNN/DiGress read `${data.noise_levels}`. Overriding via CLI produces subtly different behavior depending on which key you target.

### Moderate Friction

**3.6 `oc.env:TMGG_S3_BUCKET` in discrete diffusion logger config** causes composition failure on local machines without that env var set, even if S3 logging is not intended. Direct barrier to local testing.

**3.7 `DigressDenoisingLightningModule.sample()` uses an ad-hoc denoising loop** (interpolation between prediction and current state), not proper ancestral sampling as in `DiscreteDiffusionLightningModule.sample_batch()`. MMD results from denoising vs generative are not directly comparable, even on identical distributions. (`digress_denoising/lightning_module.py:239-314`)

**3.8 Stage definition YAML `architectures` are bare path strings with no validation.** A typo silently fails at config composition time, not at YAML load time. The `+` prefix in HP overrides is opaque without Hydra experience.

**3.9 `SpectralDenoisingLightningModule._make_model` reads from a fixed set of `self.hparams` keys.** Any model with novel architecture-specific parameters (e.g., `attention_radius`) requires modifying the Lightning module's `__init__` to accept and forward them. (`spectral_arch_denoising/lightning_module.py:128-147`)

## 4. Documentation and Docstrings

### Critical (Actively Misleading)

**4.1 `README.md` project structure references non-existent paths.** Lists `models/spectral_denoisers/` and `models/factory.py` which do not exist at the documented locations. (`README.md:191-196`)

**4.2 `gnn_denoising/README.md` uses wrong class names.** Lists `GNN_sym` and `NVGNN`; actual classes are `GNNSymmetric` and `NodeVarGNN`.

**4.3 `dataset.py:68` comment says "round-robin" but code uses `torch.randint`.** The described behavior was never implemented.

**4.4 `hybrid.py:104` comment says "apply output transformation" but code only does `bmm`.** The next comment then explains why the transformation isn't there — first comment describes something that doesn't happen.

**4.5 `wandb.yaml:7` hardcodes tags `["attention", "denoising"]`** for all runs regardless of model type. A GNN or DiGress run gets tagged "attention."

**4.6 `CassicalGraphs` class name is misspelled.** No docstrings on the class or any methods. (`classical.py`)

**4.7 `anudataset.py:257` stale TODO** — `# TODO need to add size flag for size` next to a hardcoded `5000` limit with no explanation.

### Moderate (Missing Important Info)

**4.8 `run_experiment()` has no docstring.** Central orchestration function called by every runner, handling skip logic, checkpoint resumption, final_eval gating, and W&B cleanup. (`run_experiment.py:52`)

**4.9 Mixed docstring styles.** NumPy style in most files, but Google style (`Args:`/`Returns:`) in `nvgnn.py`, `mha_layer.py`, `attention.py`, `dataset.py`, `dataset_wrappers.py`, `logging.py:93-97`. Sphinx style (`:param:`/`:return:`) in `anudataset.py`.

**4.10 `lightning_base.yaml` describes a module that may no longer be the active base.** Contains `loss_type`, `noise_type`, `noise_levels` that duplicate `_base_infra.yaml` and `base_config_training.yaml`. Unclear if legacy or active.

**4.11 `exp_configs/README.md:78-84` shows `lr: 1e-2` as default** but `_base_infra.yaml` sets `learning_rate: 1e-3`. The README shows the stage override, not the base default.

**4.12 `compute_eigenvalue_entropy` docstring says "Normalized by log(n)"** but the code does not divide by log(n). (`eigenstructure_study/analyzer.py:139`)

**4.13 `compute_spectral_distance` metric `"frobenius"` is not spectral** — it computes Frobenius norm of the matrix difference, ignoring eigendecompositions. Misleading function name for this metric option. (`eigendecomposition.py:66,87`)

**4.14 `default.yaml` logger comment says "Missing credentials will gracefully skip"** — contradicts the project's "fail loudly" principle in CLAUDE.md.

## 5. Generative Experiment Observability

### High (Affects Correctness or Comparability)

**5.1 LR scheduler silently disabled for both generative experiments.**
`scheduler_config` is defined in `_base_infra.yaml` at root level but not propagated into `config.model`. Both `GenerativeLightningModule` and `DiscreteDiffusionLightningModule` receive `scheduler_config=None` and train without LR scheduling despite the config implying cosine warmup. This is a silent behavioral divergence from intended configuration.

**5.2 MMD kernel mismatch between experiments.**
Gaussian diffusion uses `mmd_kernel: gaussian` (L2); discrete diffusion uses `gaussian_tv` (total variation). Results from the two experiments are not directly comparable. No warning is emitted. (`base_config_gaussian_diffusion.yaml:28` vs `discrete_default.yaml:13`)

**5.3 Gaussian validation evaluates at a single fixed noise level.**
`_eval_step` uses `noise_schedule[num_diffusion_steps // 2]` (line 289) — a fixed midpoint. If the model learns well at low-noise but poorly at high-noise levels, this fixed evaluation obscures the problem. Denoising experiments average over all `eval_noise_levels`.

### Medium (Reduces Diagnostic Value)

**5.4 `visualization_interval` is a dead config field** for both generative modules.
The parameter is stored as an instance attribute but never checked or used anywhere in either module. The denoising base class actively checks it and produces adjacency matrix visualizations, network graphs, and multi-noise panels. Neither generative experiment produces any visualizations during training. (`gaussian_diffusion_generative/lightning_module.py:115`, `discrete_diffusion_generative/lightning_module.py:136`)

**5.5 Node/edge loss not logged separately in discrete training.**
`TrainLossDiscrete` computes `loss_X` (node CE) and `loss_E` (edge CE, weighted by `lambda_E=5.0`) but returns only their weighted sum. The individual components are discarded before logging. Cannot distinguish whether the model underfits node types vs edge types. (`train_loss.py:64-81`)

**5.6 Novelty metric is silently skipped.**
`SamplingEvaluator.evaluate()` supports novelty via `train_graphs` parameter, but neither generative module passes training graphs. Novelty is always `None` even when `"novelty"` is in `eval_metrics`. (`sampling_evaluator.py:185-187`)

**5.7 `gaussian_emd` kernel implemented but undocumented.**
Fully functional in `mmd_metrics.py:311-360` but not listed as a valid option in either generative module's constructor type annotations or docstrings. Would work if passed as a string but is not discoverable.

### Low (Minor Gaps)

**5.8 No gradient norm logging.** `gradient_clip_val=1.0` is active but clipping events are invisible — cannot tell whether clipping fires frequently (instability) or rarely.

**5.9 No timestep distribution logging (discrete).** `t_int` is sampled uniformly but never logged. Cannot verify even coverage across the timestep range.

**5.10 No generated graph statistics alongside MMD.** Mean degree, density, connectivity fraction of generated samples are not reported. If the model collapses to empty/fully-connected graphs, MMD will be high but the cause invisible.

**5.11 VLB components not broken down per timestep.** A per-t L_t curve would show whether the model learns well at all timesteps or only at easy ones. Standard in DDPM-style analysis.

## Summary

| Category | Critical | Moderate | Minor |
|---|---|---|---|
| Math/naming | 4 | 10 | — |
| Architecture | 5 | 10 | — |
| Extensibility | 5 | 4 | — |
| Documentation | 7 | 7 | 8 |
| Observability | 3 | 4 | 4 |

The most impactful issues for immediate attention:

1. **`E[..., 0]` adjacency inversion** (1.2) — GNN projections and spectral features computed on the complement graph
2. **`p_in`/`p_out` config mismatch** (1.3) — SBM probabilities not actually configurable in discrete diffusion
3. **LR scheduler silently disabled** (5.1) — both generative experiments train without the configured cosine warmup
4. **KL prior direction** (1.1) — logged NLL values are technically wrong (low practical impact but misleading diagnostics)
5. **No visualizations in generative experiments** (5.4) — `visualization_interval` is a dead field; no sample galleries during training
6. **Two incompatible data pipelines** (3.3) — cannot share datasets between denoising and generative experiments without reimplementation
