# Bug: Modal run aborts with SIGABRT from graph-tool during Lightning sanity check

**Status:** **CONFIRMED AND FIXED** on 2026-04-15 via a Modal
differential across T4 and A10G hosts. See "Confirmation and
resolution" at the end of this document.

**Reported:** 2026-04-15
**Branch:** `fixes-2026-04-14`
**Observed with:** `DEBUG-run-upstream-digress-sbm-modal-no-viz.zsh`
**Exit code:** `-6` (SIGABRT)
**Scope:** any generative-eval-enabled training on Modal (not specific
to no-viz; see below why no-viz does not suppress it).

## Symptom

On a fresh Modal run of the upstream-style SBM DiGress training script,
the Python subprocess inside the container aborts with a C++ runtime
error immediately after Lightning prints the model parameter table:

```
terminate called after throwing an instance of 'std::length_error'
  what():  vector::_M_fill_insert
Exit code: -6 (SIGABRT)
```

Output stops between the Lightning `transfer_batch_to_device` warning
and the first training-step log, i.e. at the point where Lightning's
initial validation sanity check runs.

## Root cause

**Lightning's initial validation sanity check triggers generative
evaluation on a randomly-initialized model, and the graph-tool–based
SBM accuracy metric aborts the process.**

Three facts combine:

1. Lightning's trainer defaults to `num_sanity_val_steps=2`. It is not
   overridden in `configs/` or in
   `src/tmgg/experiments/exp_configs/base/trainer/default.yaml`, so it
   applies. Lightning calls the validation loop (including
   `on_validation_epoch_end`) **before** the first training step, with
   `self.global_step == 0`.

2. In `src/tmgg/training/lightning_modules/diffusion_module.py:531`,
   generative evaluation is gated only by:
   ```python
   if self.global_step % self.eval_every_n_steps != 0:
       return
   ```
   At sanity-check time, `global_step == 0`, so `0 % N == 0` for any
   `N`. The gate **passes**, and the evaluator runs on samples
   generated by an untrained model.

3. `GraphEvaluator.evaluate()` invokes `compute_sbm_accuracy`
   (`src/tmgg/evaluation/graph_evaluator.py:540`), which spins up a
   `concurrent.futures.ThreadPoolExecutor` (default `is_parallel=True`)
   and calls `gt.minimize_blockmodel_dl` plus a thousand
   `state.multiflip_mcmc_sweep(beta=np.inf, niter=10)` iterations per
   graph. graph-tool uses OpenMP internally and **is not thread-safe**
   under Python's `ThreadPoolExecutor`. The combination of
   concurrent MCMC sweeps with pathological (near-empty or near-
   complete) graphs from a random-init model produces an internal
   `std::length_error` during `vector::_M_fill_insert`.
   `_is_sbm_graph` only catches `ValueError`
   (`src/tmgg/evaluation/graph_evaluator.py:239`); a raw C++ abort
   escapes as `std::terminate` → SIGABRT.

### Why the Modal preflight did not catch it

`_run_config_preflight` in `src/tmgg/modal/_functions.py` exercises
datamodule setup, module instantiation, noise process init, a single
model forward, and a single loss compute. It does **not** invoke the
sampler or the evaluator. Everything that crashes here is on a path
preflight never touches.

### Why the `no-viz` wrapper does not suppress it

`DEBUG-run-upstream-digress-sbm-modal-no-viz.zsh` sets
`evaluation.visualization.enabled=false`. That only short-circuits the
`if self.visualization["enabled"]` branch at
`diffusion_module.py:547`, which is reached **after** the evaluator has
already run and returned. The metric computation is not affected.

## Evidence

- The crash message `vector::_M_fill_insert` is a libstdc++ internal —
  the fault is in an in-process C++ extension, not in Python.
- ORCA (the other C++ path) is invoked via
  `subprocess.check_output` (`src/tmgg/evaluation/orca/__init__.py:150`),
  so an ORCA crash would surface as `CalledProcessError`, not a SIGABRT
  on the parent process. ORCA is ruled out.
- The MMD pipeline uses `gaussian_tv` by default — pure numpy/scipy/
  networkx. `gaussian_emd` would pull in POT (C++) but is not on the
  default code path. MMD is ruled out.
- graph-tool remains the only in-process C++ extension reachable from
  `evaluator.evaluate()` with the default skip_metrics set (no `sbm`
  skip). The code path is `compute_sbm_accuracy` →
  `ThreadPoolExecutor.map(_is_sbm_graph, ...)` →
  `gt.minimize_blockmodel_dl` + `multiflip_mcmc_sweep`.
- The output log terminates exactly where the first
  `on_validation_epoch_end` would fire (Lightning sanity-check phase).

## Fix

Two layered changes; either alone prevents this crash, together they
also prevent a future recurrence if a second concurrency hazard is
introduced.

### Primary (cheapest, targets the symptom)

Skip generative evaluation during Lightning's sanity check. Add, at
the top of `on_validation_epoch_end` in
`src/tmgg/training/lightning_modules/diffusion_module.py` (before the
existing `eval_every_n_steps` gate):

```python
if self.trainer.sanity_checking:
    return
```

This matches the precedent already used in
`src/tmgg/training/progress.py:456,485`.

Additionally, tighten the `eval_every_n_steps` gate so it doesn't fire
at step 0: either `if self.global_step == 0 or ...` or switch to
`self.global_step > 0 and self.global_step % self.eval_every_n_steps == 0`.
A zero-step eval on an untrained model yields meaningless metrics
either way.

### Secondary (remove the underlying hazard)

Flip `is_parallel` to `False` by default in `compute_sbm_accuracy`
(`src/tmgg/evaluation/graph_evaluator.py:287`), or, better, run the
per-graph calls through a `ProcessPoolExecutor` rather than a
`ThreadPoolExecutor` (graph-tool is process-safe, not thread-safe).
Also broaden the `except ValueError` to `except (ValueError, RuntimeError,
Exception)` at
`src/tmgg/evaluation/graph_evaluator.py:239` so that graph-tool's
Python-translated errors always return `0.0` rather than propagating.
(Leave genuine C++ aborts aborting — that is signalling a real hazard.)

### Regression test

Per `CLAUDE.md` ("Whenever you fix a bug … first take the time to add
the bug as a unit test before fixing"), add a test that constructs a
`DiffusionModule` with `eval_every_n_steps=N`, puts it into a Lightning
sanity-checking state (`trainer.sanity_checking=True`), calls
`on_validation_epoch_end`, and asserts the evaluator is not invoked.
A second test should assert `compute_sbm_accuracy` is never called at
`global_step == 0`.

## Reproduction / verification path

## Notes / caveats

- graph-tool thread-unsafety is a library-level fact, not a project
  regression. The bug is latent whenever `compute_sbm_accuracy` runs
  under `ThreadPoolExecutor` on degenerate inputs.
- `global_step % N == 0` at step 0 is a separate latent bug: it means
  *every* `eval_every_n_steps`-gated path fires at step 0, not just
  this one. Worth auditing other callers for similar assumptions
  (e.g. eigenstructure logging, any `eval_every_n_steps`-gated
  callbacks).
- Local reproduction is not diagnostic here: graph-tool, ORCA, and the
  glibc/libstdc++ linkage all live in the Modal image only. Any
  host-side repro would use a different build against a different
  libc/libstdc++ and produce a different failure mode. Confirmation
  must use the Modal image — hence the skip-sbm vs skip-orbit
  differential in the verification section.

## Confirmation and resolution

The SIGABRT hypothesis was confirmed on Modal via a six-run
differential on 2026-04-15. The instrumented evaluator path (temporary
`[DEBUG-SBM]` prints) showed that (a) the evaluator does run during
Lightning's sanity check with `global_step=0`, (b) generated graphs
are not degenerate (n=20, edges in 119–189), and (c) the crash occurs
*after* `g.add_edge_list(...)` succeeds, inside graph-tool's
`minimize_blockmodel_dl` / `multiflip_mcmc_sweep` C++ routines while
multiple Python thread workers call them concurrently. Thread-mode
crashes on both A10G (`vector::_M_fill_insert`) and T4
(`malloc() unaligned tcache chunk`) — same root cause, different
surface symptom depending on which glibc/libstdc++ vector operation
raced first. Process-mode completes on both hosts.

| Run | Host | Executor | Result                                    |
|-----|------|----------|-------------------------------------------|
| A   | A10G | thread   | SIGILL on graph-tool import (transient)   |
| A2  | A10G | thread   | SIGILL on graph-tool import (transient)   |
| A3  | T4   | thread   | Completed (lucky race)                    |
| B   | T4   | process  | Completed                                 |
| C   | T4   | thread   | SIGABRT `malloc unaligned tcache chunk`   |
| D   | T4   | process  | Completed                                 |
| E   | A10G | thread   | SIGABRT `vector::_M_fill_insert` (original)|
| F   | A10G | process  | **Completed — verifies fix**              |

**Applied fix.** `compute_sbm_accuracy` in
`src/tmgg/evaluation/graph_evaluator.py` now invokes its parallel
graph-tool calls through a
`concurrent.futures.ProcessPoolExecutor(mp_context=multiprocessing.get_context("spawn"))`.
Spawn ensures each worker starts a fresh Python interpreter, importing
graph-tool from scratch with no inherited OpenMP/libstdc++ state from
the parent. Sequential mode (`is_parallel=False`) remains available
for callers that want no pool at all.

A regression test at
`tests/experiment_utils/test_graph_structure_metrics.py::TestSBMAccuracy::test_parallel_executor_is_process_pool_not_thread_pool`
patches both executor classes inside `graph_evaluator` and asserts the
process class is instantiated and the thread class is not, so any
future regression to threads fails CI loudly.

Notes on what the original bug report also flagged:

- The "generative eval runs at `global_step == 0` because `0 % N == 0`"
  observation remains technically true and still worth fixing for
  semantic reasons (evaluating an untrained model yields meaningless
  metrics). But the SIGABRT is fully resolved by the executor fix
  alone; the modulo gate is not the root cause.
- The sanity-check `if self.trainer.sanity_checking: return` guard
  suggested in the earlier "Primary fix" section is orthogonal to the
  crash. It's a reasonable cleanliness improvement; filing separately
  would be appropriate if desired.
