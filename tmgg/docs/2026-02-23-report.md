# tmgg Codebase Architecture Report

**Date:** 2026-02-23
**Scope:** Full `src/tmgg/` package — structure, factoring, abstraction hierarchy, adherence to stated engineering principles.

## Evaluation Principles

1. **Simple thing done right, then orthogonal additive features** — clean base abstractions with independent feature composition
2. **No surprises** — functions do what their names suggest, no hidden side effects
3. **One way of doing things** — a single canonical path for each operation
4. **No fallbacks or backwards compatibility** — fail loud, no graceful degradation, no legacy shims
5. **Good ML research engineering** — clean separation of config → data → model → training → analysis

---

## Executive Summary

The core training loop (`run_experiment` → `DenoisingLightningModule` → `DenoisingModel`) and the spectral model family are well-designed. The base abstractions are narrow, the config hierarchy is clean, and the model registry pattern works. Where problems concentrate is in the *orchestration and infrastructure layers*: multiple redundant entry points for the same operation, a fully-built coordinator that nothing uses, a Hydra launcher with a sequential-dispatch bug, several backwards-compatibility shims that violate the project's own "fail loud" policy, and layer inversions where `experiment_utils` imports upward from `experiments`.

Thirty findings are catalogued below, classified as GOOD (preserve), SMELL (mild), PROBLEM (fix), or DEAD (delete).

---

## 1. Models and Data

### What works well

The three-tier model hierarchy (`BaseModel` → `DenoisingModel` → concrete) is narrow and correct. `DenoisingModel` defines `forward(x, t)`, `transform_for_loss`, `logits_to_graph`, `predict`. The spectral denoiser family adds `_spectral_forward(V, Lambda, A)` as a clean template method — each spectral model (LinearPE, Bilinear, FilterBank, SelfAttention) implements only this one function and composes orthogonally on the base.

`MODEL_REGISTRY` with `@register_model` in `factory.py` replaces what would otherwise be a long if-chain. It fails loudly on duplicate registration.

`AdjacencyBatch` and `CategoricalBatch` in `data/batches.py` are frozen dataclasses with eager shape validation and immutable operations. This is the right pattern for tensor containers.

## 2. Experiment Utilities and Experiments

### What works well

`run_experiment()` is a clean single-purpose function with explicit dependency order: seed → dirs → checkpoint → data → model → callbacks → loggers → trainer → fit → test → eval → sync. Each concern is delegated to a named helper. The function has no hidden state.

The config inheritance chain (`_base_infra.yaml` → `base_config_training.yaml` → per-experiment config) separates infrastructure, denoising-specific, and model-specific concerns correctly.

Steps-based configuration throughout (`max_steps`, `val_check_interval`, `every_n_train_steps`) is consistent with the project principle.

### Problems

| # | Finding | Location | Class |
|---|---------|----------|-------|
| 13 | **Seven per-experiment runner files are identical boilerplate.** Each is a `@hydra.main` wrapper that calls `run_experiment(cfg)` with a different `config_name`. The only meaningful variation is `skip_final_eval=True` for generative experiments, which could be a config flag. One factory + a table in `pyproject.toml` would replace all seven. | `experiments/*/runner.py` | PROBLEM |
| 14 | **Two systems for running sweeps exist in parallel.** Path A: `ExperimentCoordinator.run_stage()` via `experiments/stages/runner.py` with `sweep=true`. Path B: `--multirun` via `TmggLauncher`. Both dispatch to `run_experiment()` but through different checkpoint, output directory, and resumption mechanisms. Neither path uses the other's state. | `experiment_utils/cloud/coordinator.py`, `hydra_plugins/tmgg_launcher/launcher.py` | PROBLEM |
| 15 | **`ExperimentCoordinator` appears unused in practice.** The coordinator, `StageConfig`, `StageResult`, `CloudStorage`, `S3Storage`, `LocalStorage` form ~700 lines of coordination infrastructure. The actual workflow uses `--multirun` or per-experiment CLIs directly. The coordinator's `_check_completed()` only checks its own `LocalStorage`, not Lightning checkpoints. | `experiment_utils/cloud/coordinator.py`, `experiment_utils/cloud/storage.py` | DEAD |
| 16 | **`check_wandb_run_exists()` silently returns False on all errors.** A network timeout or bad credentials would cause experiments to run that should be skipped, potentially duplicating hours of GPU work. Violates the project's "fail loud" principle. Should distinguish `CommError` from unexpected exceptions. | `experiment_utils/run_experiment.py:106-113` | PROBLEM |
| 17 | **`load_checkpoint_with_fallback()` is a compatibility shim.** Its fallback path introspects `__init__` signatures and silently drops unknown hyperparameters. The function name advertises the fallback. | `experiment_utils/checkpoint_utils.py:18-116` | PROBLEM |
| 18 | **Eigenstructure and embedding studies each have two CLIs** — a Click interface and a Hydra runner exposing identical functionality. A user must know which to use. | `experiments/eigenstructure_study/cli.py` vs `runner.py`, `experiments/embedding_study/cli.py` vs `runner.py` | PROBLEM |
| 19 | **Logger config parsing is a hand-rolled dispatch loop** instead of using Hydra's `_target_` instantiation (which the trainer, model, and data module all use). Every new logger backend requires a new `elif`. | `experiment_utils/logging.py:99-185` | SMELL |
| 20 | **`generate_run_id()` hardcodes field awareness** — special-cases `k`, `diffusion_steps`, `learning_rate`, `weight_decay`, `_target_`. Any new hyperparameter requires editing this function. | `experiment_utils/run_experiment.py:59-82` | SMELL |
| 21 | **`sanity_check.py` uses name-based model dispatch** — `"Attention" in model.__class__.__name__` to choose between eigenvector and adjacency inputs. Fragile, duplicates logic already in `DenoisingLightningModule`. | `experiment_utils/sanity_check.py:395-406, 535-548` | SMELL |

## 3. Modal and Hydra Launcher

### What works well

The core Modal pattern is sound: `_functions.py` is the single deployment boundary, `ModalRunner` serializes configs to YAML and dispatches via `Function.from_name()`, and `modal` is imported lazily everywhere except `_functions.py`. The transport layer carries no experiment logic.

`modal/app.py` cleanly separates GPU tier constants from the SDK. `modal/confirmation.py` is a simple append-only JSONL log.

### Problems

| # | Finding | Location | Class |
|---|---------|----------|-------|
| 22 | **`TmggLauncher.launch()` calls `run_experiment()` sequentially** in a for loop, negating Modal's parallelism. `ModalRunner.run_sweep()` correctly uses `starmap` for parallel dispatch, but the Hydra launcher never calls it. | `hydra_plugins/tmgg_launcher/launcher.py:80-103` | PROBLEM |
| 23 | **`modal/evaluate.py` breaks the transport abstraction** by directly importing model classes, data modules, and metrics. It reconstructs a data module by manually parsing OmegaConf config, duplicating what the CLI path already does. ~213 lines of evaluation logic that should flow through the CLI. | `modal/evaluate.py:111-119, 346-381` | PROBLEM |
| 24 | **GPU tier selection logic duplicated in three places** — `_functions.py:317`, `runner.py:154`, and `cli/evaluate_mmd.py:40`. If a tier is added, one of three will be missed. | `modal/_functions.py`, `modal/runner.py`, `modal/cli/evaluate_mmd.py` | PROBLEM |
| 25 | **`ModalRunner()` in `stages/runner.py` ignores `stage_config.gpu_type`** — always constructs with the default (`"debug"`). A sweep configured for A10G silently runs on T4. | `experiments/stages/runner.py:65-67` | PROBLEM |
| 26 | **Top-level `ModalRunner` import in `experiments/stages/runner.py`** triggers Modal SDK initialization even for local runs. Should be a lazy import inside the `if run_on_modal` branch. | `experiments/stages/runner.py:25` | PROBLEM |
| 27 | **`_cli_cmd` defaults to `"tmgg-discrete-gen"`** if absent from config. Silent fallback to the wrong experiment type. Should raise. | `modal/runner.py:197, 283, 340` | SMELL |
| 28 | **`parallelism: int = 4` in launcher config is never read.** The launcher iterates sequentially. | `hydra_plugins/tmgg_launcher/config.py:35` | DEAD |
| 29 | **Four unreferenced volume utility functions** (`ensure_dataset_cached`, `_cache_pyg_dataset`, `list_cached_datasets`, `clear_outputs_volume`). Dataset caching strategy that was never wired up. | `modal/volumes.py:81-159` | DEAD |

## 4. Analysis and Cross-Cutting

### What works well

The analysis package correctly accesses experiment results only through the filesystem (safetensors, JSON, W&B parquets). The registry + plugin pattern for reports is clean. The config hierarchy is documented and non-redundant.

### Problems

| # | Finding | Location | Class |
|---|---------|----------|-------|
| 30 | **`spectral_deltas.py` imports from `experiments/`** — a hard layer inversion. `compute_laplacian` is a pure ~30-line math utility that has no business in `experiments/eigenstructure_study/`. It forces `experiment_utils` to reach upward. | `experiment_utils/spectral_deltas.py:18, 108` | PROBLEM |
| 31 | **Duplicate `compute_subspace_distance`** with different APIs and return types. `experiment_utils/metrics.py:86` (single matrix, returns float) vs `experiments/eigenstructure_study/analyzer.py:257` (batched tensors, returns Tensor). Both exported from their respective `__init__.py`. | `experiment_utils/metrics.py`, `experiments/eigenstructure_study/analyzer.py` | PROBLEM |
| 32 | **Eigenstructure report has a placeholder/fallback mode** — generates synthetic data and charts when the results directory is missing, instead of raising. Directly violates the "no graceful fallback" principle. | `analysis/reports/eigenstructure.py:45-128` | PROBLEM |
| 33 | **`analysis/__init__.py` triggers heavy imports** via side-effectful report registration. Importing any parsing function loads matplotlib, seaborn, and the full plotting stack. | `analysis/__init__.py:23` | SMELL |
| 34 | **`tmgg/__init__.py` eagerly imports `experiment_utils` and `models`**, coupling the package namespace to torch and pytorch_lightning. An analysis script needing only `parse_stage` loads the entire training stack. | `tmgg/__init__.py:7` | SMELL |
| 35 | **`matplotlib.use("Agg")` set in three places** (`logging.py:10`, `experiment_utils/plotting.py:10`, `experiment_utils/sanity_check.py:10`) at import time, and once inside a function (`analysis/figures.py:46`). | various | SMELL |
| 36 | **Analysis CLI has no entry point in `pyproject.toml`** — reachable only via `python -m tmgg.analysis.cli`. | `pyproject.toml` (absence) | SMELL |

## Architectural Diagram

```
Desired layering:

  data  →  models  →  experiment_utils  →  experiments  →  analysis
                              ↑                  |
                              |                  |
                              +------ (VIOLATION: spectral_deltas imports from experiments)

  hydra_plugins  →  experiment_utils/cloud  →  modal/
                                                  ↑
                                                  |
                              (VIOLATION: experiments/stages/runner.py imports ModalRunner eagerly)
                              (VIOLATION: modal/evaluate.py imports model/data classes)
```

No circular dependencies exist (the graph is acyclic), but three upward dependency violations break the intended layering.

## Priority Actions

### High priority (architectural violations, silent bugs)

1. **Fix the Hydra launcher sequential dispatch** (#22) — this is a correctness bug that makes Modal sweeps run sequentially instead of in parallel.
2. **Fix `ModalRunner()` ignoring `gpu_type`** (#25) — sweeps silently use the wrong GPU tier.
3. **Move `compute_laplacian` out of `experiments/`** (#30) — resolves the layer inversion.
4. **Consolidate GPU tier selection** (#24) — extract to a single function in `modal/app.py`.
5. **Make `check_wandb_run_exists` fail loud on unexpected errors** (#16) — catch only `wandb.errors.CommError`, re-raise everything else.

### Medium priority (redundancy, dead code)

6. **Delete `ExperimentCoordinator` and its storage layer** (#15) — ~700 lines of unused infrastructure.
7. **Delete the four unreferenced volume functions** (#29).
8. **Collapse the seven identical runner files** (#13) into one factory.
9. **Choose one CLI per study** (#18) — keep the Hydra runner, remove the Click CLI (or vice versa).
10. **Finish or revert the `PlaceHolder` → `CategoricalBatch` migration** (#1).
11. **Delete continuous diffusion dead code** (#10) and `CategoricalBatch.from_placeholder` (#11).

### Low priority (smells, polish)

12. Remove the `nvgcn_layer.py` backwards compatibility shim (#3).
13. Remove `assume_adjacency_input` deprecated parameter (#4).
14. Extract eigendecomposition pad-to-k as a shared utility (#5).
15. Make `tmgg/__init__.py` imports lazy (#34).
16. Move `models/embeddings/` into `experiments/embedding_study/` (#9).
17. Register `tmgg-analysis` entry point (#36).
